# Stack Knowledge

Accumulated findings on technology stacks. This file is an input to the
Tech Stack Evaluation (theme 20 of the onboarding interview). Each
evaluation consults it and appends what it learned, so the knowledge
compounds across projects.

The file ships with general baseline notes. To make findings available to
future projects, copy useful entries back into the template.

## How to use

- Before an evaluation, read the entries for the candidate stacks.
- After an evaluation, append or update an entry with what was observed —
  community size, documentation quality, ecosystem maturity, tooling, the
  observed token-cost tendency, and the setup recommended for the stack
  (MCP servers, skills, tools).

## Baseline notes

- Large, mainstream stacks (for example TypeScript/Node, Python) have very
  large communities, abundant documentation and mature tooling — typically
  the lowest token cost and the most reliable implementation.
- Smaller or fast-moving stacks may serve a project well, but tend to cost
  more tokens and iteration, because community knowledge and documentation
  are thinner. Flag this to the PO; it is a trade-off, not a veto.
- Strong static typing, fast linters and clear error messages shorten the
  feedback loop and make implementation more reliable, regardless of
  community size.
- A stack finding is tied to a model version and a framework version. Date
  every entry and note the model it was observed with — it has a shelf life.
