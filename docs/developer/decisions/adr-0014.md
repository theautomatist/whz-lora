# ADR-0014 — LoRaWAN Network Server: ChirpStack v4

- Status: Accepted
- Date: 2026-05-26

## Context

The whz-lora project (concept paper Section 1) needs a LoRaWAN Network
Server (LNS) at its core. The stack constraints fixed during onboarding
Cluster A are:

- Self-hosted, on-premise (no cloud LNS).
- Reusing existing open-source software rather than building one.
- Docker Compose as primary runtime, single host.
- EU868 radio band.
- Must talk to a Kerlink Wirnet iFemtoCell Evolution 868 (PDTIOT-IFE03)
  over the Semtech UDP Packet Forwarder or Basics Station protocol.
- External consumers subscribe via MQTT; no own application UI is built.
- Per-device payload decoding (JavaScript codecs).
- Verification check requires injection of a synthetic uplink via a
  Gateway Simulator.

## Decision

Adopt **ChirpStack v4** as the LoRaWAN Network Server, deployed as the
official `chirpstack-docker` Compose stack (ChirpStack, Gateway Bridge
UDP, Gateway Bridge Basics Station, Mosquitto, PostgreSQL, Redis,
REST API proxy).

## Rationale

The evaluated alternative was **The Things Stack Open Source (CE)**.
Reasons for choosing ChirpStack v4:

- **Out-of-the-box localhost setup**: `git clone && docker compose up`
  is functional without TLS, domain or secrets pre-configuration. TTS CE
  requires generating secrets, editing `ttn-lw-stack-docker.yml`, DB
  init, and effectively a TLS endpoint for the UI to be usable
  (1–2 hours setup vs. <30 minutes).
- **Kerlink iFemtoCell support**: ChirpStack ships a dedicated `.ipk`
  package for the iFemtoCell — `chirpstack-concentratord_4.7.1-r1_klkgw.ipk`
  — with explicit installation documentation; both Packet Forwarder and
  Basics Station are present in the default Compose. TTS CE has no
  equivalent first-class artifact for this specific gateway model. The
  Concept Audit (research recheck, 2026-05-26) confirmed the package
  exists and is documented; whether the *Evolution* sub-model
  (PDTIOT-IFE03) is fully covered by the same `.ipk` must be confirmed
  at the first F-0001 directive (the documentation names only the base
  iFemtoCell).
- **MQTT simplicity**: Embedded Mosquitto with the standard
  `application/{id}/device/{eui}/event/up` topic schema and plain
  username/password auth. TTS CE requires API-key generation and
  `@tenant`-prefixed identifiers for MQTT clients, which complicates the
  external research-subscriber path.
- **Verification fit**: ChirpStack v4 exposes a stable gRPC API on port
  8080 that lets a smoke-test script provision tenant, application,
  device profile and device, then inject a synthetic uplink via the
  Packet Forwarder UDP socket; the resulting event is visible on the
  embedded Mosquitto broker — exactly the verification check committed
  to in concept paper Section 3 (`brocaar/chirpstack-simulator` was
  considered and rejected as the inject path; see ADR-0015). No
  comparable simulation path for TTS CE.
- **Knowledge availability**: Larger amount of community Q&A, tutorials
  and DeepWiki coverage on the public web — lower expected token cost
  for the research agent over the life of the project.

## Consequences

- The runtime is multi-container (~7) but each is small and the upstream
  Compose is treated as the single source of truth — own changes are
  layered as overrides, not forks.
- PostgreSQL and Redis are first-class dependencies; the project commits
  to PostgreSQL as the LNS-internal store (the SQLite alternative
  surfaced during interview Cluster B is incompatible with v4 and is
  dropped from the concept paper).
- Default MQTT auth is broker-level Mosquitto credentials; F-0003
  acceptance criteria are tied to that mechanism.
- The Gateway Bridge installation on the Kerlink uses the upstream
  `.ipk` package; firmware/KerOS compatibility for the iFemtoCell
  Evolution must be confirmed during the first F-0001 directive.
- Switching the LNS later would invalidate codecs (JS dialect), MQTT
  topic schema and the simulator — i.e. it is the least-reversible
  decision in this project, hence captured here.
