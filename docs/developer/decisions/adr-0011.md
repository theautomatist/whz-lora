# ADR-0011 — Conversational Intake

- Status: Accepted
- Date: 2026-05-19

## Context

ADR-0002 defined the directive lifecycle with "Intake — the PO files the
directive as a GitHub issue" as step 1 and "Triage & Plan" as step 4. Two
problems surfaced. First, a sequencing error: triage classifies a directive
as lightweight or full, but a lightweight classification is meant to
collapse the specification — which, at step 4, has already run. The
classification must precede the specification. Second, requiring the PO to
author issues against a template makes the GitHub issue an input the PO
must produce, when the natural interface is a conversation.

## Decision

- The main session is the PO's single point of contact. The PO describes
  intent in plain language; the main session clarifies it in a short
  dialogue, classifies it, and records it as a GitHub issue. The issue is
  an output of that documentation work, not a form the PO fills. The PO
  may still file a simple issue directly.
- Triage moves into step 1 ("Intake & Triage"). The classification steers
  the specification forward: a lightweight task gets a one-line spec from
  the main session; a full task gets a `spec-analyst` specification.
- Step 4 becomes "Plan" (build planning only). The lifecycle keeps nine
  steps.
- A feature is not defined via an issue: its definition lives in the
  feature registry entry. The intake conversation routes feature work to
  the feature process; the issue created there is only the build vehicle.
- Between Gate 1 and Gate 2 the team works autonomously — the main session
  returns to the PO only at the two gates or to escalate.

## Rationale

- Triaging at intake removes the sequencing contradiction: classification
  precedes specification, which it governs.
- A conversational interface matches how a PO actually works and lets the
  main session illuminate the request technically, so subagents receive a
  precise, self-contained directive.
- Naming autonomy explicitly turns "the team works on its own between the
  gates" from an implicit hope into a stated contract.

## Consequences

- The PO interacts by conversation; the main session operates GitHub.
- The directive issue template is filled by the main session (or the PO
  directly), not authored by the PO against a rigid form.
- ADR-0002 still holds (the lifecycle and its tooling); this ADR refines
  its intake and triage.
