# ADR-0018 — Gateway management over USB-NDIS, with Windows-host firewall rules

- Status: Accepted
- Date: 2026-05-26

## Context

The first directive against feature F-0001 (`feat: bring Kerlink
iFemtoCell Evolution online with ChirpStack v4`, issue #3) had to
choose a physical and IP-level topology between the gateway and the
Docker-hosted ChirpStack stack.

The Kerlink Wirnet iFemtoCell-evolution exposes three management
paths:

1. **Ethernet** on `eth0` — the production path; the gateway gets a
   DHCP lease from the campus network, the stack runs somewhere
   reachable.
2. **USB-Ethernet (RNDIS)** on the USB-C port — Kerlink-provisioned
   private subnet `192.168.120.0/24`, gateway at `.1`, attached host
   at `.31`. Always available, no LAN configuration required.
3. **Cellular** on `wwan0` — out of scope.

For project bring-up and lab work, the host is the developer's
Windows laptop running Docker Desktop, not a Linux server with
Ethernet to the campus.  The host gets a different IP each time it
joins a different LAN; the gateway can't be configured to follow it.

## Decision

Use the **USB-NDIS path for gateway management and packet-forwarder
traffic** until the project moves the stack onto a stationary host
(WHZ lab VM or dedicated Linux server).  The procedure is:

- The gateway's USB-C port is plugged into the host running Docker
  Desktop.  Windows enumerates a `Remote NDIS Compatible Device`;
  the IP topology is fixed: gateway `192.168.120.1`, host `…31`.
- `/etc/lorafwd.toml` on the gateway is configured with
  `node = "192.168.120.31"`.
- The host's Windows Firewall is opened for inbound UDP 1700 (packet
  forwarder) and ICMPv4 (diagnostics) on the `Any` profile, via two
  `New-NetFirewallRule` calls run from an elevated PowerShell.

## Rationale

- **No LAN required.**  A developer can bring up an end-to-end LoRa
  pipeline against the stack on a flight, in a hotel room, or
  unplugged from the campus network.  The USB cable both carries data
  and powers the gateway during bring-up.
- **Determinism.**  The IPs are baked into the Kerlink firmware and
  do not change between hosts; the smoke-test procedure and the
  bring-up doc can hard-code them.
- **Docker Desktop on Windows publishes container ports to all host
  network interfaces by default.**  Once the firewall is open, UDP
  packets reaching the host on the USB-NDIS adapter are forwarded
  through Docker Desktop's NAT into the `chirpstack-gateway-bridge`
  container with no further configuration.
- **Symmetry with SSH and the web UI.**  Both management paths use
  the same RNDIS link; one cable, one IP, one mental model.
- **Reverting later is cheap.**  Switching to Ethernet on a lab VM
  is a single `node = "<lab-host-ip>"` change in `lorafwd.toml`.

## Consequences

- **Firewall rules are part of host setup.**  Two
  `New-NetFirewallRule` invocations must run once per host.  The
  procedure is documented in
  [docs/user/kerlink-ifemtocell-bring-up.md](../../user/kerlink-ifemtocell-bring-up.md),
  Step 2.  Without these rules the entire pipeline is silent; this is
  the most common failure mode we observed.
- **`Public` network classification is fine.**  Group Policy on
  WHZ-managed PCs prevents non-admin users from changing the USB
  adapter's network category to `Private`; the firewall rules use
  `-Profile Any` to apply regardless of category.
- **Bandwidth is small but symmetric.**  Stats frames are sub-kB at
  30-second intervals; LoRa uplinks are even smaller and bursty.
  USB-Ethernet (CDC-NCM at full-speed USB 2.0) handles this with
  orders of magnitude to spare.
- **Single point of contact.**  The gateway is *only* reachable from
  the one PC it is plugged into.  Other team members cannot SSH in,
  nor can a continuous-integration runner.  That is fine for the
  current single-operator scope; when the project gains a second
  human operator or a CI hardware-in-the-loop runner, the gateway
  moves to Ethernet (separate directive).
- **The CI pipeline is unaffected.**  CI continues to exercise the
  *simulator* path of F-0001 via `scripts/smoke_test.py`; the real
  gateway is exercised only in manual local runs.

## Open questions for later

- When the project moves onto a fixed Linux host (no Docker Desktop,
  native Docker engine), the firewall stops being a manual step.
  Re-visit and likely supersede this ADR at that point.
- For multi-gateway rollouts, the USB-NDIS path scales only to as
  many gateways as the host has USB ports.  Ethernet on a dedicated
  segment will be the right answer.
