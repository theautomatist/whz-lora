# ADR-0003 — Project Onboarding Interview

- Status: Accepted
- Date: 2026-05-18

> Note: the output model described below was later refined — the concept
> paper replaces `setup/project-profile.md` and a status field replaces the
> file-existence signal (see ADR-0006), and a skill entry point was added
> with the interview now at 22 themes (see ADR-0007).

## Context

The template is project-agnostic. When it is cloned for a concrete project,
the tech stack, data structures, interfaces and many other specifics are
still open. They must be captured before reliable work can start, and the
agents and `CLAUDE.md` must be tuned to them.

## Decision

A first-run onboarding interview is added. `setup/interview.md` holds a
themed question catalog (20 themes in 4 clusters) and the interview
protocol. On first start — detected by the absence of
`setup/project-profile.md` — the main session conducts the interview,
co-develops data structures and flows with the Product Owner, writes
`setup/project-profile.md`, fills the Project-Specific section of
`CLAUDE.md`, tunes the agents and quality gates, and seeds the initial
documentation.

## Rationale

- The signal "not yet onboarded" is the absence of a file, not a flag or a
  hook — the simplest possible mechanism (KISS).
- A documented catalog makes the interview reproducible across projects
  while staying adaptable; the protocol says to skip what does not apply.
- Tuning the agents per project is what turns the generic template into a
  project-specific team.

## Consequences

- Every project starts from a consistent, thorough profile.
- The interview is a one-time cost at project start; afterwards the
  directive lifecycle takes over.
- `setup/interview.md` must be kept current as the team or the pillars
  evolve.
