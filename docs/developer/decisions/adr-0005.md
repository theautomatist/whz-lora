# ADR-0005 — Tech Stack Evaluation in Onboarding

- Status: Accepted
- Date: 2026-05-18

## Context

When a project has no mandatory tech stack, the choice must be made well —
it is the least-reversible decision in the project. The right choice
depends on many project specifics (data, processing, performance,
deployment, infrastructure) that only become clear during the onboarding
interview.

## Decision

- The onboarding interview's tech-stack theme is split. Theme 3 (early)
  captures only constraints and givens — mandated or forbidden
  technologies, existing systems, team expertise, preferences. The
  evaluation and decision move to theme 20, near the end of the interview.
- Theme 20 runs only if the stack is open. The `research` agent evaluates
  2-4 candidate stacks against a rubric: project fit, implementation
  effort, community size & knowledge availability, library availability,
  tooling & feedback quality.
- Instead of an unmeasurable "AI performance" score, the evaluation uses
  community size and knowledge availability as a measurable proxy, and
  presents a token-cost advisory: stacks with thinner documentation and
  community typically cost more tokens and iteration.
- Findings accumulate in `setup/stack-knowledge.md` across evaluations.
- The recommendation requires PO sign-off and is recorded as an ADR.

## Rationale

- Placing the decision at the end means it is informed by all prior
  answers, which strongly constrain the stack.
- Capturing constraints early (theme 3) is still necessary — a fixed stack
  changes how everything else is discussed.
- "Claude Code performance per stack" has no reliable public data;
  community size and documentation quality are honest, measurable proxies
  and also signal the likely token cost.
- Conceptual themes (architecture, data model) are discussed
  stack-agnostically first; theme 22 revisits them for stack-specific
  detail once the stack is chosen.

## Consequences

- Open stacks get an informed, evidence-backed, PO-approved decision.
- Architecture and data modeling may need a brief stack-specific revisit
  after the decision (handled in theme 22).
- `setup/stack-knowledge.md` must be carried back into the template to
  benefit future projects — a manual step, consistent with the accepted
  template-drift trade-off (ADR-0001, ADR-0002).
