# ADR-0008 — Feature Lifecycle and Registry

- Status: Accepted
- Date: 2026-05-18

## Context

The directive lifecycle handles the work of a change, but nothing held a
standing, structured description of what the product does. GitHub issues
are transient, ADRs record decisions, and the concept paper is the initial
definition frozen at onboarding. There was no single source of truth for
the product's features.

## Decision

- A feature registry is introduced as the single source of truth for the
  product's features — `docs/developer/features.md`, growing into a
  `features/` folder past about eight features.
- A feature process (`setup/feature-process.md`, the `/feature` skill)
  governs adding, changing and removing features: classify, short feature
  interview, impact analysis, registry draft, PO sign-off, hand to the
  directive lifecycle, registry sync on close.
- The main session proposes the feature process when a request clearly
  concerns a feature; minor work stays a plain directive.
- A removed feature keeps its entry, marked `removed`, with the reason.
- The onboarding seeds the initial registry from the concept paper.

## Rationale

- A registry kept honest by the lifecycle gives a durable answer to "what
  does this product do" that issues and ADRs cannot.
- The feature process wraps the directive lifecycle rather than replacing
  it (KISS) — it adds feature-level structure and reuses the build process.
- The dynamic layout avoids both an unwieldy single file and premature
  folder structure.

## Consequences

- Feature-affecting work is not done until the registry reflects it (added
  to the Definition of Done).
- Traceability completes: feature ↔ directives ↔ ADRs ↔ commits.
- The registry must be migrated as it grows; the feature process does this.
