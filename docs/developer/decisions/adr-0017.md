# ADR-0017 — Verification runs locally; CI deferred until self-hosted GitLab move

- Status: Accepted
- Date: 2026-05-26

## Context

`CLAUDE.md` (Testing & Definition of Done) and the concept paper §3
(Quality & Testing) treat the GitHub Actions `verify` job as the
canonical, automated verification gate for every directive: stack up,
smoke test, teardown — all on a clean Ubuntu runner per PR.

The first directive (#1, "bring up ChirpStack v4 Docker Compose stack
with verification smoke-test") exposed a hard blocker on this gate:
GitHub Actions on `theautomatist/whz-lora` (GitHub Free, private repo)
stopped firing workflow runs on the PR branch after the initial
failed runs. Three push events (`b55a856`, `fffa5e8`, `ad51a7a`) did
not register as `pull_request synchronize` events, the manual
`workflow_dispatch` call returned HTTP 500, and a PR close/reopen
cycle had no effect. The most likely root cause is the GitHub Free
Actions-minutes quota on private repositories being exhausted; a
transient GitHub-side delivery glitch is the secondary hypothesis.
Either way, the project cannot ship directives that depend on this
gate as long as the situation persists.

The PO has decided to **move the project off GitHub onto a
self-hosted GitLab instance** later, where the WHZ has its own
runners and no minute quota. Until then, work continues on this
repository for code hosting and review, but the verification gate is
moved off CI.

## Decision

Verification runs **locally on the developer host** until the GitLab
migration. The canonical verification check stays the same end-to-end
flow — Docker Compose stack up → smoke-test exercises gRPC,
Mosquitto, UDP inject, MQTT receive — but the runner is the developer
machine, not GitHub Actions.

Mechanically:

- `.github/workflows/ci.yml` stays in the repository, as the future
  basis for the GitLab CI port. It is no longer the gate. Anyone may
  push without waiting for a green CI status.
- The Definition of Done item "CI passing" is rewritten as **"local
  verification check passing"**, evidenced by a verbatim paste of the
  smoke-test exit message in the pull-request review.
- The Gate-2 PO acceptance procedure picks up the verification
  evidence from the PR conversation rather than from a CI status
  check.
- The directive lifecycle is otherwise unchanged: branch + draft PR,
  reviewer report, BLOCKER-cleared, Gate 2.

## Rationale

- The verification check itself is intact — it is a local Python
  script against a local Compose stack. Nothing depends on the
  runner.
- The GitLab move is a project decision already taken; building a
  workaround on GitHub Actions (Pro upgrade, public repo, etc.)
  would burn time for a transient setup.
- Keeping the workflow file in place means the GitLab move is a
  port, not a rewrite. The bash steps in `.github/workflows/ci.yml`
  map line-by-line to a `.gitlab-ci.yml` job.

## Consequences

- Loss of the automatic regression net per PR until the GitLab move.
  Reviewer pickiness and local-verification discipline replace it.
- The `verify` status check is no longer a required check (irrelevant
  while branch protection is also deferred — see ADR-0016 — but
  worth recording).
- Anyone running the verification check needs Docker Desktop, Python
  3.12+, and Node.js locally. This is now an explicit prerequisite
  (CLAUDE.md "Project-Specific" updated).
- Branch protection on `main` (ADR-0016) and CI-on-GitHub (this ADR)
  are both deferred to the GitLab move. The two ADRs share the same
  superseding event.
- This ADR's status changes to `Superseded` when the GitLab move
  brings runners back online and verification re-automates.
