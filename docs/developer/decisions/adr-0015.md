# ADR-0015 — Verification Toolchain: Python Smoke-Test via gRPC + UDP

- Status: Accepted
- Date: 2026-05-26

## Context

The whz-lora verification check (concept paper Section 3) requires an
automated end-to-end run that exercises the Docker Compose stack: spin
it up, provision a Tenant + Application + Device Profile + Device,
inject a synthetic uplink as if it had arrived from a real gateway, and
confirm the decoded event appears on the embedded MQTT broker. This
must run unattended in the GitHub Actions CI job inside under 10
minutes.

Two candidate tools were considered during onboarding (Theme 20) and
re-examined in the Concept Audit (Step 8):

1. `brocaar/chirpstack-simulator` — an existing simulator originally
   chosen as part of the stack evaluation rationale.
2. A small project-built Python script using the official
   `chirpstack-api` gRPC library plus a raw UDP socket against the
   Packet Forwarder port.

The Concept Audit surfaced two findings against the simulator:

- The repository, while claiming v4 compatibility, has had no commits
  since November 2023 (~2.5 years stale at time of writing).
- Even with the simulator, the CI job still needs a separate
  provisioning step (Tenants/Apps/Devices/Codecs do not exist on a
  fresh stack), so the simulator does not fully solve the problem
  alone.

## Decision

Build the verification step as a single Python 3.12 script
(`scripts/smoke_test.py`) that:

1. Connects to the ChirpStack gRPC API on `localhost:8080` using the
   `chirpstack-api` Python package, with an admin API key passed via
   `.env`.
2. Creates a known test Tenant, Application, Device Profile and Device
   (idempotently — on second run, find-or-create rather than fail).
3. Constructs a Semtech UDP Packet Forwarder protocol `PUSH_DATA`
   packet with a stats frame followed by a synthetic uplink for that
   Device's DevEUI, and sends it to `localhost:1700` over UDP.
4. Subscribes to `application/<id>/device/<eui>/event/up` on the
   Mosquitto broker using `paho-mqtt` and waits up to 30 seconds for
   the matching event to arrive.
5. Returns exit code 0 on success, non-zero with a clear error
   message on failure.

The script does **not** use `brocaar/chirpstack-simulator`. The
required Python packages are `chirpstack-api` and `paho-mqtt`; both are
on PyPI and pinned in `scripts/requirements-test.txt` once the script
ships.

## Rationale

- The provisioning step is unavoidable; once a provisioning script
  exists, the additional code to also inject a UDP frame is small
  (tens of lines).
- `chirpstack-api` is maintained as part of the ChirpStack project
  itself — its release cadence tracks the LNS we depend on, removing
  the stale-tool risk that disqualified the simulator.
- A single Python script is easier to debug in CI than orchestrating a
  separate simulator container plus a provisioning script.
- The same script doubles as the documentation example for
  "how to talk to this stack from outside" — useful for onboarding
  future researchers writing MQTT consumers.

## Consequences

- `scripts/smoke_test.py` becomes the canonical entry point of the
  verification check; the CI workflow runs exactly it.
- `chirpstack-api` is pinned to a version compatible with the
  ChirpStack server version in the Compose; both must move together.
- The script depends on stable details of the Packet Forwarder
  protocol — a 2014 Semtech specification that has not meaningfully
  changed; low risk.
- If a Basics Station path is later needed for verification, that is a
  separate, smaller directive (additional inject mode in the same
  script).
- Acceptance criterion in F-0002 ("each codec has a test suite under
  `codecs/`") is unaffected by this decision and runs as `node --test`
  in CI, independent of the Python smoke-test.
