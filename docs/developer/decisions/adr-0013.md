# ADR-0013 — Branch, Merge and Gate-2 Mechanics

- Status: Accepted
- Date: 2026-05-19

## Context

ADR-0002 defined the directive lifecycle and ADR-0011 refined intake to a
conversation, but the GitHub mechanics — branching, PR review, merge
strategy, Gate-2 capture — were not pinned down. The lifecycle hand-waved
"the pull request merges" and "Gate 2 is the PO's acceptance" without
saying how those happen in practice. With the autonomy contract ("between
the gates the team works alone") the operational rails need to enforce
themselves rather than rely on convention.

## Decision

- **One branch and one PR per directive.** Branch name mirrors the
  Conventional Commits type + the issue number: `feat/123-add-login`,
  `fix/124-typo`, `docs/125-update-readme`.
- **The main session creates the branch and the draft PR at the start of
  Step 4 (Plan)** — directly after Gate 1, before any code is written.
- **Squash & merge.** The resulting main-branch commit follows
  Conventional Commits with `closes #N` in the footer. Merged branches are
  auto-deleted.
- **Branch protection on main** is set during onboarding (interview
  Step 7): a PR is required, status checks must pass, direct pushes are
  forbidden.
- **Gate 2 = PR-Approval.** The PO approves the pull request — branch
  protection makes the approval technically necessary before merge. The
  Gate-2 checkbox on the directive issue is removed; the PR is the single
  locus for delivery acceptance. Gate 1 remains an issue checkbox (no
  branch exists yet at that point).
- **CI failures during Build are BLOCKERs** like any reviewer finding and
  count toward the three-round Build cap.
- **The reviewer's report is posted by the main session as a PR comment**,
  so the report persists on GitHub and the implementer can answer
  line-by-line. Aggregate counters land in the issue's process log at
  Close.
- **Lightweight directives skip the reviewer.** The main session checks
  the diff itself before merge; squash & merge still applies.
- **Status labels map to lifecycle stages:** `spec` covers Steps 2–3,
  `building` covers Steps 4–5, `verify` covers Steps 6–7–8; `blocked`
  replaces the current stage label on escalation; a closed issue is done.

## Rationale

- A per-directive branch with squash-merge is the standard GitHub flow
  that best matches Conventional Commits (one logical change = one commit
  on main) and gives the reviewer a coherent diff.
- Putting Gate 2 on the PR removes a duplicated entry — branch protection
  enforces it; the issue's checkbox would be redundant. Gate 1 stays at
  the issue because the branch does not exist that early.
- Counting CI failures toward the Build cap keeps "round" as a single
  notion across reviewer findings, quickcheck, and CI.
- Reviewer-comments-on-PR uses GitHub's native review surface and
  survives the chat session — a next session can re-read the report.

## Consequences

- `directive.md` no longer carries a Gate-2 checkbox; it points to the
  linked PR.
- Onboarding's Step 7 includes the branch-protection and merge-strategy
  configuration as required setup.
- The implementer's commits during Build can be casual (squash collapses
  them); the *PR* final message — and therefore the main commit — must be
  Conventional-Commits-clean.
