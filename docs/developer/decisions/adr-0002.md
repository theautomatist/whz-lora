# ADR-0002 — Directive Lifecycle and Tooling

- Status: Accepted
- Date: 2026-05-18

> Note: intake and triage were later refined — the main session is the PO's
> conversational point of contact, and triage moves into step 1 (see
> ADR-0011).

## Context

The initial setup (ADR-0001) covered building a task well but left both
ends of the work open: how a Product Owner directive enters the system and
how a result is accepted. A productive, reliable PO-driven system needs a
defined lifecycle, verification against intent, and documentation for both
developers and end users.

## Decision

- A directive lifecycle is introduced: intake, specification, PO sign-off
  (Gate 1), triage & plan, build, verify, document, PO acceptance
  (Gate 2), close.
- Directives are tracked as GitHub issues; traceability comes from issues
  linked to commits and pull requests.
- A fourth agent, `spec-analyst`, turns directives into testable
  specifications with acceptance criteria.
- Two Product Owner gates are mandatory: spec sign-off before building and
  delivery acceptance at the end.
- Documentation is split into two separate sites: developer documentation
  and user documentation.
- Continuous integration runs on GitHub Actions as a verification gate.

## Rationale

- GitHub issues over OpenProject — the backlog must be machine-accessible
  for the agents; the `gh` CLI and ecosystem make this trivial, and
  traceability to commits is free. OpenProject would add a server and a
  custom integration for little extra value at this stage (KISS).
- A `spec-analyst` is the minimum needed to make "the PO only gives
  directives" actually work — fuzzy directives must become testable specs.
- Two gates trade a little speed for high reliability: the PO confirms
  intent before work and the result after.
- No technical-writer or security-auditor agent yet — the main session and
  the reviewer cover these for now; add them only when they earn it (KISS).

## Consequences

- The PO has two clear touch points per directive and explicit control.
- Every directive is traceable end to end via its issue.
- Two documentation sites mean two build targets and slightly more upkeep.
- The system depends on GitHub for issues and CI.
