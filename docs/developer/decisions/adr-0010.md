# ADR-0010 — Extensible Team with Specialist Agents

- Status: Accepted
- Date: 2026-05-19

## Context

The base team (ADR-0001, ADR-0002) is four generalist agents — analysis,
building, review, research. Projects differ widely: one ships a rich user
interface, another is firmware on hardware, a third is regulated. A fixed
generalist team either lacks domain depth where a project needs it or
carries breadth no project uses. Onboarding theme 22 already asked "is a
further agent justified", but with no procedure — the bar and the how were
undefined, so the question was easy to answer carelessly in either
direction.

## Decision

The four agents are a fixed base team, present in every project. Beyond
them, a project may add **specialist agents** through a defined "Extending
the Team" procedure in `CLAUDE.md`.

- A specialist is a specialization of one of four archetypes — analyst,
  builder, controller, researcher — and inherits that archetype's tool and
  model profile. It is not a freely invented role.
- A specialist is added only when all three hold: the need is recurring,
  it requires competence the base team genuinely lacks, and a persistent
  agent beats the cheaper alternatives (a review dimension on `reviewer`,
  guidance in an existing agent's prompt, a one-off `research` brief).
- The main session assesses this at onboarding (interview theme 22) and
  during the feature process (Step 3, when a feature introduces a new
  domain). The PO signs off; the addition is recorded as an ADR and the
  team table in `CLAUDE.md` is updated.
- A specialist that stops earning its place is removed.

## Rationale

- Archetype-bound — a specialist is a known shape, not an open-ended role;
  tools and model follow the archetype, so least privilege and cost
  discipline hold without a separate judgement each time.
- High bar with cheaper rungs first — keeps KISS: most domain needs are met
  by a review dimension or agent guidance; a new agent is the exception.
- Two assessment points — onboarding catches the project's known domains;
  the feature process catches domains that surface only later.
- PO sign-off and an ADR — a team change is as visible and traceable as any
  architecture decision.

## Consequences

- The team fits the project without bloating every project with unused
  agents.
- Specialists live in the instantiated project, not the template; like all
  agent definitions (ADR-0001) they do not propagate back — accepted
  deliberately.
- The procedure must be applied with discipline; without the three-part
  test, "add an agent" becomes the easy and usually wrong default.
