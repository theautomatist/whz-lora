# ADR-0004 — Testing and Release Verification

- Status: Accepted
- Date: 2026-05-18

> Note: the fixed deployment smoke test was later generalised to a
> per-project verification check, so the lifecycle fits non-service
> artifacts too (see ADR-0012).

## Context

The setup had a CI gate but did not define what is tested, how deployment
is verified, or how the user documentation is kept correct. A productive
system must verify both the software and that a real user can set it up
from the user documentation alone.

## Decision

- Testing follows the test pyramid (unit, integration, end-to-end) and runs
  in ephemeral Docker containers via CI; this is the deep, developer-facing
  verification.
- The Verify stage of the directive lifecycle adds a deployment smoke
  test: the built software must deploy and start cleanly in a container.
- A release procedure is added. Before every release the main session runs
  a user-documentation walkthrough: in a fresh, empty container, using only
  the user documentation, it sets up and operates the software; any gap is
  a documentation defect to fix.
- Releases use semantic versioning and a `CHANGELOG.md`.

## Rationale

- Docker over VMs for test isolation — clean, fast, reproducible; VMs only
  where a full OS is genuinely needed (see ADR-0001).
- The clean container makes the documentation walkthrough honest: missing
  dependencies or commands fail visibly.
- The walkthrough runs as a main-session procedure, not a separate agent
  (KISS); a dedicated doc-verifier agent can be added later if it earns it.
- The walkthrough is tied to releases, not every directive, to balance cost
  against keeping the user documentation correct.

## Consequences

- The user documentation is verified end to end before each release.
- An AI walkthrough is not a true beginner; an occasional real-user test
  remains valuable.
- Releases gain a small, defined pre-flight procedure.
