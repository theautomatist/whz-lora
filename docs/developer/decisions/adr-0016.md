# ADR-0016 — Branch Protection on main: deferred

- Status: Accepted
- Date: 2026-05-26

## Context

`CLAUDE.md` (Process & Standards, Directive Lifecycle Step 9) requires
branch protection on `main` that enforces a passing CI run and a
mandatory pull request before any merge. The GitHub API call to install
that protection (`PUT /repos/{owner}/{repo}/branches/main/protection`)
fails with HTTP 403 on `theautomatist/whz-lora`:

> Upgrade to GitHub Pro or make this repository public to enable this feature.

The repository is private on a GitHub Free account; branch protection
is therefore not available through the technical platform alone.

Three options were considered: (1) make the repository public,
(2) upgrade the account to GitHub Pro, (3) leave the repository private
and operate without enforced protection.

## Decision

Adopt option (3): leave the repository private, operate without
GitHub-enforced branch protection. The Gate-2 PR review and the CI
green requirement remain as **process discipline**, documented in
`CLAUDE.md` and exercised through the directive lifecycle, but they
are not enforced by the platform on this repository.

## Rationale

- whz-lora is a one-person project (PO + AI team); there is no second
  human pushing to `main` whose mistakes the protection would catch.
- The CI workflow (`.github/workflows/ci.yml`) still runs on every
  push and pull request; visibility of failures is preserved even
  without enforcement.
- Making a research-stage repository public to unlock a single
  policy feature is a disproportionate trade-off — the project's
  configuration may still contain LoRaWAN-network details (Device
  EUIs, gateway IDs, network identifiers) the PO prefers to keep
  unpublished until the first features are stable.
- A paid GitHub Pro subscription for one feature is not justified at
  this stage.

## Consequences

- Direct pushes to `main` are technically possible; the team's
  contract is to merge only via PR and only when CI is green.
- The Definition of Done in `CLAUDE.md` (CI passing, reviewer report
  clear, etc.) is unchanged — only its mechanical enforcement is
  removed.
- Revisit when: the repository becomes public, the project moves to
  an organisation account with team accounts, or a second human
  contributor joins. At that point the standard branch-protection
  setup should be applied — the documented `gh api -X PUT
  /repos/.../branches/main/protection` command from the onboarding
  hand-off is the right entry point.
- This ADR's status changes to `Superseded` (linking the new ADR)
  when branch protection becomes active.
