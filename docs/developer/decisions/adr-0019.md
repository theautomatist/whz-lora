# ADR-0019 — Own provisioning app for actuators: a narrow exception to Non-Goal #5

- Status: Accepted
- Date: 2026-06-12

## Context

The Product Owner has a set of LoRaWAN actuators (Stellantriebe, heating/valve
type) with their OTAA credentials and an operational problem: the DevEUI and
AppKey are long hex strings that get mistyped when entered by hand, devices end
up not joined, and it is noticed too late. The desired remedy is a
**mobile-capable web app** to enrol devices (single form and CSV bulk) and,
crucially, get **direct feedback whether each device has actually come online**.

This collides with the documented foundation:

- Concept paper **Non-Goal #5**: *"Keine eigene Anwendungs-UI über die vom LNS
  mitgelieferte UI hinaus."* (Section 2 Output repeats *"keine eigene
  Anwendungs-UI in dieser Iteration"*.)
- [ADR-0014](adr-0014.md) set the ChirpStack browser UI as the administration
  surface; feature F-0002 frames device management as happening there *"ohne
  eigene UI zu bauen"*.
- The concept paper's *Build-it-ourselves* section, however, already
  pre-authorised a gRPC provisioning script *"sobald mehr als 2–3 Devices"*.

The concrete design was reached in a full grilling session with the PO on
2026-06-12 and signed off (Gate 1 of issue #6, feature F-0005).

## Decision

Build a small, single-container web app (feature **F-0005**) as a **narrow,
explicitly documented exception** to Non-Goal #5, and amend the concept paper
accordingly.

Shape of the app:

- **Three functions:** single-device form; CSV bulk import (one row per device);
  a commissioning **dashboard** showing a three-state signal per device —
  ⚪ Provisioned → 🟡 Joined → 🟢 Online (first real uplink).
- **Reuses the proven gRPC core**: provisioning logic is the find-or-create
  pattern from `scripts/smoke_test.py`, factored into
  `scripts/chirpstack_client.py` and shared with the app; it adds the OTAA
  device-keys step the smoke test never exercised
  (`DeviceService.CreateKeys`/`UpdateKeys`).
- **No own database** — ChirpStack/PostgreSQL stays the single source of truth;
  the device list and live status are read back from ChirpStack on demand.
- **gRPC, not the REST proxy** — ChirpStack's own recommendation; avoids adding
  a service.
- **Security posture:** reachable only inside the WHZ network / VPN, **not**
  exposed to the public internet — the same trust boundary as the existing
  ChirpStack UI on `:8080`. Optional HTTP Basic auth is available
  (`PROVISIONING_AUTH_USER` / `PROVISIONING_AUTH_PASS`); when unset the app runs
  open for local testing and logs a warning.

Concept-paper amendment — Non-Goal #5 and the Section 2 Output line are changed
from an absolute prohibition to the narrow exception: *a mobile-capable
provisioning and commissioning app for OTAA actuators, WHZ-net/VPN-only, without
its own persistence and without re-implementing existing LNS functions.*

## Rationale

The value the PO needs is **not delivered by the stock ChirpStack UI** for a
fleet workflow: typo-resilient entry, and a consolidated **commissioning
confirmation** (defined as *first uplink received*, the honest "it really
works" signal). This solves a recurring operational failure, not a cosmetic
preference.

Alternatives considered:

- **ChirpStack UI only** — rejected: it is the status quo that *causes* the
  mistyping and late-detection pain and gives no consolidated join/uplink
  feedback across a batch.
- **Headless CLI / file manifest** — considered (and pre-authorised by the
  concept paper) and rejected as the *primary* surface: the PO's real,
  recurring need is mobile, on-site enrolment with immediate join feedback,
  possibly by non-expert staff — which a terminal does not serve. The
  provisioning *core* is identical either way; the web app is a thin shell over
  the same shared `chirpstack_client.py`, so little is lost by skipping a
  separate CLI deliverable.
- **REST proxy (`chirpstack-rest-api`)** — rejected: gRPC is ChirpStack's
  recommended path and avoids adding a seventh service. (ADR-0014 lists the
  REST proxy as part of the upstream stack, but this repo deliberately does not
  run it.)

KISS is preserved: one container, no database, no frontend framework
(server-rendered HTML), reusing proven code.

On security: a **write surface for OTAA root keys must not be weaker than the
rest of the stack** — F-0003 requires authentication even for read-only MQTT
subscribers. Hence the optional Basic auth and the explicit WHZ-net/VPN-only
posture. Public-internet exposure is deliberately out of scope and would be a
separate decision (TLS termination, hardened auth). A later QR/OCR scanner will
need a secure context (TLS) for browser camera access — noted for the roadmap.

## Consequences

Positive:

- Actuators are enrolled with far less risk of a mistyped EUI/AppKey, and an
  installer can confirm on-site (over VPN) that a device joined and is sending.
- The OTAA `CreateKeys` path — never exercised before (the smoke test uses ABP)
  — is now covered by the extended verification.
- The gRPC provisioning core is shared between the smoke test and the app
  (`scripts/chirpstack_client.py`), reducing duplication.

Negative / trade-offs:

- Non-Goal #5 is no longer absolute; it now carries one narrow, named
  exception. This ADR is the record of why.
- A long-running HTTP service must be operated and kept version-aligned with
  ChirpStack (`chirpstack-api` pinned to 4.18.0).
- Without Basic auth configured, anyone who can reach the app inside the
  WHZ-net/VPN can provision or delete devices. Accepted within the trusted
  boundary; mitigated by the optional auth and by not publishing the app to the
  public internet.

Roadmap, explicitly deferred (not in F-0005 v1): QR-code scanner, OCR label
recognition, MQTT live-push auto-refresh, room/apartment assignment, Class C /
mains devices, public-internet exposure, per-user authentication.

This ADR narrows the "no own UI" framing of [ADR-0014](adr-0014.md) for this one
case only; ADR-0014 otherwise stands.
