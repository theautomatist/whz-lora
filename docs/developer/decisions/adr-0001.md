# ADR-0001 — AI Development Team Setup

- Status: Accepted
- Date: 2026-05-18

> Note: the team described below was later extended — ADR-0002 introduces a
> fourth agent, `spec-analyst`, for the specification step.

## Context

The project is built by an AI development team. It needs a reusable base
that is strategically identical across all projects and only takes in the
project-specific content.

## Decision

The base rests on five pillars: Process & Standards, Documentation System,
AI Team, Quality Assurance, Infrastructure. The team consists of three
agents: `implementer` (builder, Sonnet), `reviewer` (controller, Opus,
read-only) and `research` (researcher, Sonnet, parallelizable). Building
and control are strictly separated; the main session orchestrates. KISS is
the overriding principle for all processes and structures.

## Rationale

- Sonnet for building and research — capable enough and clearly cheaper;
  Opus judgment is only needed for the critical review.
- reviewer without write access — forces findings to go to the main
  session instead of being silently fixed.
- One `research` agent instead of several — different "departments" come
  from different briefs, not duplicate files (KISS).
- Ready-made MCP servers / CLIs instead of custom builds — less development
  and maintenance effort.
- Docker instead of a VM pool — cheaper, faster test isolation.
- No dedicated commit agent — a documented commit standard in CLAUDE.md
  achieves uniform, reproducible commits without an extra moving part (KISS).

## Consequences

- The team is low-cost and lean to maintain.
- Deep branching is only possible as a wave model steered by the main
  session, since subagents cannot start subagents.
- Agent definitions live in the template; improvements do not propagate
  automatically into already-instantiated projects — accepted deliberately.
