# ADR-0009 — Multi-Repository Projects

- Status: Accepted
- Date: 2026-05-18

> Note: sub-repository instantiation was later hardened — the pristine
> template carries a `.template-marker` file, and `scripts/init-project.ps1`
> refuses to run from any source without it. This prevents accidentally
> cloning an already-instantiated repository (for example the main
> repository) as the source for a sub-repository.

## Context

With coding agents, a product can be larger than one team could build and
may split into separate, independently deployable packages (for example a
frontend and a backend). The template assumed one repository per project.

## Decision

- A product may span several repositories: one main repository and one or
  more sub-repositories. Every repository is its own template instance and
  is consistent in itself.
- The main repository coordinates. It holds a manifest (`repositories.md`),
  the product-level feature registry, the contracts between
  sub-repositories, and product-level ADRs and docs — linked, not copied.
- Repositories are linked by the manifest and cloned as siblings; git
  submodules are not used.
- Features follow a two-level model: product features in the main
  repository, component features in the sub-repositories.
- The main repository is onboarded first and decides the split; each
  sub-repository is then onboarded with a scoped interview. The tech stack
  is chosen per sub-repository.
- A cross-repository directive is a parent issue in the main repository
  with linked child issues in the sub-repositories.

## Rationale

- A repository is the context boundary an agent works in; several focused
  repositories suit agent-driven development better than one large monorepo
  context. Split only at genuine package boundaries.
- A manifest avoids the well-known submodule footguns for agents and humans.
- The main repository owning the cross-repository contracts prevents
  interface drift — the main risk of a multi-repository setup.

## Consequences

- Multi-repository products gain a coordination layer; single-repository
  projects are unaffected.
- The main repository's coordination artifacts are created during its
  onboarding, not shipped in the template.
- Sub-repositories onboard separately, inheriting the product context.
